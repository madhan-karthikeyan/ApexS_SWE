Everything you need to go from zero to a deployed, research-grade sprint planning system. Architecture decisions, technology choices, API design, and a step-by-step development roadmap.

01
High-Level System Design
Five-layer pipeline — data flows in, intelligence flows out
▸ Presentation Layer
Dashboard
Dataset Upload
Sprint Config
Sprint Plan View
Explainability Panel
Plan Approval
↕ REST API (JSON / HTTPS)
▸ Application Layer
Context Extractor
Weight Learning (scikit-learn)
Optimization Engine (PuLP ILP)
Explainability Engine (SHAP)
Celery Task Queue
↕ SQLAlchemy ORM / MinIO SDK
▸ Data Layer
PostgreSQL 15
Redis 7 (Cache + Queue)
MinIO (File Storage)

Key principle: Each layer only speaks to its adjacent layer. The Optimization Engine runs asynchronously via Celery — the frontend polls for results so the UI never blocks.

Technology Stack
Every tool chosen for a specific reason — no cargo culting
Frontend
React 18 + TS
Component reuse, type safety, huge ecosystem for data tables & charts
Zustand
Lightweight state for sprint plan workflow (no Redux boilerplate)
shadcn/ui
Unstyled primitives → full control over UI matching Pencil mockups
React Query
Auto-polling for optimization job status (2s refetch interval)

Value vs Risk scatter chart, capacity bars, velocity trend
Vite
Sub-second hot reload for rapid UI development
Backend
FastAPI
Async-native, auto OpenAPI docs, perfect for ML pipeline integration
PuLP + CBC
Free, open-source ILP solver — handles your knapsack + dependency constraints exactly
scikit-learn
Ridge Regression learns team-specific prioritization weights from sprint history
SHAP
Model-agnostic feature attribution for explainability output
Celery + Redis
Async job queue — optimization runs in background worker, UI polls status
SQLAlchemy 2
ORM with Alembic migrations — version-controlled database schema
Data & Storage
PostgreSQL 15
ACID-compliant, array columns for story dependencies, full-text search
Redis 7
Celery broker + API response caching for fast repeated queries
MinIO
Self-hosted S3 for CSV/JSON dataset uploads — no AWS costs for lab
Infrastructure
Docker Compose
Single command spins up all 6 services — reproducible for every team member
GitHub Actions
Auto-run pytest on every push — catches regressions immediately
Railway.app
Free tier deployment with one-click PostgreSQL + Redis — no DevOps needed
Swagger UI
Auto-generated from FastAPI — team can test all endpoints without Postman
03
Core Data Flow
How a click becomes an optimized sprint plan
1
User uploads historical sprint CSV
POST /api/v1/datasets/upload → validate schema → store MinIO → save metadata to DB
↓
2
User configures sprint (goal, capacity, risk threshold, skills)
POST /api/v1/sprints/ + POST /api/v1/stories/ (bulk)
↓
3
User clicks "Generate Sprint Plan" → triggers async Celery job
POST /api/v1/plans/generate → returns job_id immediately
↓
4
Celery Worker: ContextExtractor → WeightLearning → ILP Solver → ExplainabilityEngine
Each step updates state: 20% → 40% → 60% → 80% → 100%
↓
5
Frontend polls job status every 2 seconds
GET /api/v1/plans/status/{job_id} → {status, progress, step}
↓
6
Plan ready → fetch plan + explanations → render Sprint Plan screen
GET /api/v1/plans/{plan_id} + GET /api/v1/plans/{plan_id}/explain
↓
7
User approves → exports Jira-compatible CSV
PUT /api/v1/plans/{plan_id}/approve → POST /api/v1/plans/{plan_id}/export
KEY API ENDPOINTS
Method Endpoint Purpose
POST/api/v1/datasets/uploadUpload & validate CSV dataset
POST/api/v1/sprints/Create sprint with configuration
POST/api/v1/plans/generate🔑 Trigger async optimization pipeline
GET/api/v1/plans/status/{job_id}Poll optimization progress (2s interval)

GET/api/v1/plans/{plan_id}/explainAll story explanations
GET/api/v1/plans/{plan_id}/explain/{story_id}Single story explanation
PUT/api/v1/plans/{plan_id}/approveMark plan as approved
POST/api/v1/plans/{plan_id}/exportDownload Jira-compatible CSV
GET/api/v1/reports/{team_id}/metricsVelocity, risk trend, plan quality

P1
Phase 1: Data Ingestion & Context Extraction
Week 2 · Upload CSV → validate → extract context → learn weights
TASKS
Data Pipeline Tasks — Click to expand
5 tasks · Dataset upload API, ContextExtractor, WeightLearningModel, Upload UI
WEEK 2
  • 1.1
    Define expected CSV schema: story_id, title, story_points, business_value, risk_score, required_skill, sprint_id, sprint_completed, depends_on
  • 1.2
    Build POST /datasets/upload endpoint: Pydantic validation → pandas read → MinIO store → DB metadata save → return preview (first 10 rows)
  • 1.3
    Write ContextExtractor service: compute team velocity, completion rate, skill distribution, avg risk tolerance from historical data
  • 1.4
    Write WeightLearningModel: Ridge Regression on (story_points, business_value, risk_score) → normalized weights (urgency, value, alignment)
  • 1.5

✅Milestone: A CSV file can be uploaded, validated, previewed, and the API returns a Context object with learned urgency/value/alignment weights.
P2
Phase 2: Core ILP Optimization Engine
Week 3 · The heart of the research contribution
TASKS
ILP Solver Tasks — Click to expand
4 tasks · PuLP ILP, 4 constraints, Celery async, Polling API, Processing UI
WEEK 3
  • 2.1
    Implement OptimizationEngine with PuLP: binary decision variables x_i, objective function (weighted value), 4 constraints: C1=Capacity, C2=Risk threshold, C3=Skill compatibility, C4=Dependencies
    Core formula: maximize Σ(w_value·v_i + w_urgency·(10-sp_i) + w_align·(1-r_i))·x_i subject to all 4 constraints
  • 2.2
    Wire optimization into Celery async task: ContextExtractor → WeightLearning → OptimizationEngine pipeline with progress updates at each step
  • 2.3
    Build POST /plans/generate (returns job_id) + GET /plans/status/{job_id} polling endpoint with progress percentage and current step name
  • 2.4
    Build Optimization Processing UI: animated progress bar, step checklist (matching Figure 4 in prototype), auto-redirect when done
    React Query usePlanStatus hook with refetchInterval: 2000ms
✅Milestone: Clicking "Generate Sprint Plan" runs ILP. Processing screen shows real-time steps. Optimal story list is saved to DB with correct constraint enforcement.
P3
Phase 3: Explainability Engine
Week 4 · Why was each story selected or rejected?
TASKS
Explainability Tasks — Click to expand
3 tasks · ExplainabilityEngine, explanation API, Explainability Panel UI
WEEK 4
  • 3.1
    Write ExplainabilityEngine: for each selected story compute value_contribution, risk_impact, alignment_score, confidence_score + build natural language reason string. For each rejected story, determine rejection cause (risk/capacity/skill/priority)
  • 3.2
    Build GET /plans/{plan_id}/explain (all stories) + GET /plans/{plan_id}/explain/{story_id} (single story) endpoints. Return structured JSON: {storyId, reason, contributionBreakdown, confidenceScore, is_selected}
  • 3.3
    Build Explainability Panel UI: story selector dropdown, reason card, contribution breakdown bar chart (Recharts), confidence score meter, Value vs Risk scatter plot (D3), rejected stories panel with reasons
✅Milestone: Every selected and rejected story has a human-readable explanation. Confidence scores computed. Explainability Panel renders all charts correctly.
