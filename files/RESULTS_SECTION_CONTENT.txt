4. RESULTS AND ANALYSIS

4.1 Objective
This section reports measurable outcomes of the Explainable Context-Aware Sprint Planning System and evaluates whether the implementation satisfies practical planning requirements: constrained selection quality, transparency of decisions, and end-to-end usability.


4.2 Experimental Setup
System configuration used for result generation:
- Frontend: http://localhost:5173
- Backend: http://localhost:8000
- API docs: http://localhost:8000/docs
- Infrastructure: PostgreSQL, Redis, MinIO, Celery worker (Docker Compose)

Primary dataset profile:
- File: tmp/converted_sprint_history_dataset_1.csv
- Total stories: 170
- Sprint history: 12 sprints
- Skills represented: Backend, Frontend, Database, Testing, DevOps
- Risk score range: 0.05 to 0.95
- Business value range: 5.0 to 10.0
- Completion labels: sprint_completed = 1 (119), sprint_completed = 0 (51)

Setup rationale:
- The dataset includes both completed and non-completed stories, enabling context extraction and weight learning from historical outcomes.
- Multi-sprint coverage allows trend-based reporting (velocity and value evolution).
- Skill and dependency fields enable full validation of optimizer constraints.

4.4 Constraint Satisfaction Result
The optimization engine enforces four mandatory constraints:
- Capacity constraint: total selected story points <= configured team capacity
- Risk constraint: stories exceeding configured risk threshold are excluded
- Skill constraint: stories requiring unavailable skills are excluded
- Dependency constraint: a story is selected only if prerequisite dependencies are also selected

Observed behavior across runs:
- Lower risk thresholds remove high-risk candidates from the final set.
- Lower sprint capacity reduces selected story count and total accepted scope.
- Dependency-linked stories are selected only when parent requirements are satisfied.
- Skill mismatch stories are consistently rejected under constrained skill availability.

Interpretation:
- The resulting plans are not only high-value selections, but also operationally feasible under explicit team constraints.

4.5 Explainability Result
Explainability output is generated at story level for all planning decisions:
- Selected stories include reason text, confidence, and contribution-related fields.
- Rejected stories include explicit rejection reasons (risk, skills, dependencies, capacity, or lower relative priority).

Observed quality indicators:
- No empty explanation output for non-empty plans.
- Story-level reasons are visible in both UI and API responses.
- Explanation content aligns with optimizer constraints and selected objective signals.

Interpretation:
- The planner provides decision traceability, which improves stakeholder trust and supports plan-review discussions.

4.6 Modification and Re-Optimization Result
The system supports what-if analysis by re-running optimization from an existing plan with updated constraints.

Observed behavior:
- Modify operation returns a new job_id.
- Re-run produces a new plan_id.
- Stricter constraints generate a smaller, tighter story set.

Interpretation:
- The workflow supports iterative planning and scenario comparison, which is essential in real sprint planning cycles.

4.10 Final Interpretation
The implemented system successfully transforms multi-sprint historical data into explainable sprint plans under realistic planning constraints. Results indicate that:
- planning decisions are constraint-consistent,
- explainability is generated for both selected and rejected stories,
- re-optimization enables practical what-if analysis,
- approval and export complete the full decision-support loop.

Overall, the project demonstrates a complete, usable, and evaluation-ready sprint planning workflow suitable for final review and report submission.


4.11 Limitations and Validity Notes
- Dataset realism: the current primary dataset is structured and suitable for system validation, but broader external validity should be strengthened with additional real-world public Jira datasets.
- Metric completeness: final manuscript quality improves significantly when explicit MAE, R2, and scenario-wise comparison tables are reported rather than only descriptive trends.
- Environment reproducibility: dependency parity across machines/containers should be controlled to ensure test reruns are directly comparable.
